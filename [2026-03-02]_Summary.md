# 오늘 작업 요약 (2026-03-02)

## 1. 플랫폼별 네트워크 아키텍처 분할 완료
Windows(IOCP)와 Linux(Epoll)의 동작 방식 차이를 수용하여, `Session`과 `Listener`, 그리고 `NetCore`의 소스 코드를 `Platform/Windows`와 `Platform/Linux` 폴더로 물리적으로 분리함. 가상 함수 및 Pimpl 패턴 없이 컴파일 타임에 최적의 객체가 바인딩되도록 구조를 개편함.

## 2. NetCore 통합 및 플래그 기반 설계
기존의 `IocpCore`와 `EpollCore`를 단일 `NetCore` 클래스로 통합함. 객체 생성 시점에 플랫폼 전용 플래그(`_nativeFlags`)를 설정하여, `NetCore`가 객체의 타입(Listener/Session)을 몰라도 올바른 권한으로 Epoll에 등록할 수 있도록 유연성을 확보함.

## 3. 리눅스 하이브리드 이벤트 감시 적용
- **Listener (EPOLLEXCLUSIVE)**: 다중 스레드 환경에서 여러 스레드가 동시에 `accept`를 호출하여 발생하는 경중 현상(Thundering Herd)을 방지하기 위해 `EXCLUSIVE` 플래그를 도입함.
- **Session (EPOLLONESHOT)**: 한 번에 한 스레드만 특정 세션의 데이터를 처리하도록 보장하여 데이터 순서 뒤섞임 및 버퍼 오염을 방지함. 처리 완료 후 `Update` 메서드로 감시를 재활성화(Re-arm)하는 구조를 구축함.

## 4. Scatter-Gather I/O (writev) 및 Edge-Triggered 수신 완비
- 송신: `_sendQueue`에 쌓인 다수의 버퍼를 `writev` 시스템 콜을 통해 한 번의 컨텍스트 스위칭으로 전송하도록 `FlushSend` 로직을 구현함. 부분 전송(Partial Send) 상황을 대비하여 `_sendOffset` 관리 로직을 추가함.
- 수신: Edge-Triggered(ET) 모드에 맞춰 `EAGAIN`이 반환될 때까지 `while` 루프를 돌며 모든 데이터를 긁어오도록 `ProcessRecv`를 구현함.

## 5. 더미 클라이언트 에코 테스트 및 패킷 프레이밍
더미 클라이언트/서버를 통해 실제 에코(Echo) 송수신 테스트를 진행함. 스트림 기반 TCP 특성상 발생하는 패킷 뭉침(Aggregation)을 해결하기 위해 `OnRecv`에서 정확히 `sizeof(EchoPacket)` 단위로 끊어 읽고 남은 길이를 반환하는 프레이밍(Framing) 로직을 완성함.

- **테스트 결과**: 5초당 약 2,000만 개의 패킷을 수신하며(루프백 환경), 평균 지연시간(RTT)이 약 7.8ms로 매우 안정적이고 강력한 성능을 증명함.

## 내일 할 일 (Next Steps)
- **SendQueue 자료구조 변경**: `writev`를 더 효율적으로 지원하기 위해 `Session`의 `_sendQueue`를 `std::queue`에서 `std::deque` 혹은 커스텀 링 버퍼로 변경하여 버퍼 반복 접근성을 높임.
- **패킷 직렬화 및 버퍼 풀링**: 실제 게임 데이터를 담기 위해 `SendBufferChunk`를 활용한 메모리 풀링 및 Protobuf 연동 등을 검토함.
